# PRISM Risk Scoring Algorithm

This document breaks down the mechanical logic behind the PRISM risk aggregator. 

## 🧠 Core Philosophy
Evaluating code merges is a profoundly high-stakes environment. **PRISM explicitly rejects black-box Machine Learning models for primary risk classification.** We believe that if a system flags a Merge Request as "High Risk," it must be able to justify exactly *why* structurally using auditable, deterministic data points. 

Using opaque weights generated by standard ML transformers destroys trust. Maintainers require explainability to act confidently. Therefore, PRISM computes a rigorous 0-100 score utilizing 6 heavily weighted deterministic signals directly extracted from Git and the Abstract Syntax Tree (AST). 

*Note: The LLM integration via Groq Llama-3 (the Summary Agent) is strictly utilized at the tail-end of the pipeline to synthesize these hard numbers into conversational English, not to make the underlying security decisions.*

---

## 📊 The 6 Scoring Factors

The scoring algorithm assigns points against a maximum budget of **100 Points**. Standard threshold mappings deduce the severity bands automatically.

### Factor 1: Merge Request Size (Max 15 Points)
**Philosophy**: The probability of introducing a defect scales non-linearly with the raw number of lines changed. Massive pull requests are structurally impossible for human reviewers to analyze comprehensively.
- **>500 lines**: 15 points
- **>200 lines**: 10 points
- **>50 lines**: 5 points
- **<50 lines**: 0 points
*Example*: A 650-line feature branch immediately triggers a 15-point penalty, serving as a backpressure mechanism encouraging developers to deploy smaller, iterative MRs.

### Factor 2: File Churn Volatility (Max 20 Points)
**Philosophy**: Modules that are modified continuously (churning) are empirically known defect hotspots representing shifting architectural boundaries.
- **>10 commits/30 days**: 20 points
- **>5 commits/30 days**: 12 points
- **>2 commits/30 days**: 5 points
*Example*: If a developer modifies `login_flow.py`, and `git log` reveals that file has been rewritten 14 times this month, PRISM flags it as highly unstable (+20 points).

### Factor 3: Core Module Touched (Max 20 Points)
**Philosophy**: Not all files hold equal blast potentials. Altering a CSS payload is categorically safer than modifying the OAuth lifecycle. 
- **Critical (auth, payment, security)**: 20 points
- **Core (base, common, middleware)**: 10 points
*Example*: Any path containing the substring `payment/stripe_router.py` will guarantee the PR lands on senior security desks.

### Factor 4: Testing Discrepancies (Max 20 Points)
**Philosophy**: Deleting tests or authoring business-logic without corresponding asserts is a near-guarantee of future regressions.
- **Tests removed**: 20 points
- **No new tests added**: 8 points
- **Tests present**: 0 points
*Example*: A PR adding a new data pipeline lacking any structural `pytest` integrations incurs a fast penalty.

### Factor 5: Dependency Graph Depth (Max 15 Points)
**Philosophy**: Utilizes the AST parser (`tree_sitter`) and Breadth-First-Search (BFS) traversal. We measure how many distinct structural layers break downstream.
- **Penalty Mapping**: Minimum (`Impact Depth * 5`, 15)
*Example*: If modifying `database.py` inherently breaks 3 child architectures routing to standard APIs, PRISM catches the 3-layer depth dynamically (`3 * 5 = 15 points`).

### Factor 6: Author Experience Gap (Max 10 Points)
**Philosophy**: Extracted natively via `git blame`. A seasoned maintainer modifying a core system is significantly safer than an intern modifying the identical exact system for their first time. 
- **< 3 Prior explicit commits to the touched module**: 10 points
*Example*: If User A has 54 total project commits, but *zero* inside the `/billing` directory they are currently editing, they trigger the experience gap flag (+10 points).

## 📈 Severity Mappings

| Score Range | Severity Label | Resulting Action |
|-------------|----------------|------------------|
| **0 - 30** | `LOW` | Minimal automated comments. Ideal state. |
| **31 - 60** | `MEDIUM` | Standard PRISM summary generated natively. |
| **61 - 80** | `HIGH` | Highly visible yellow label warning on Gitlab. |
| **81 - 100**| `CRITICAL` | Aggressive `high-risk` labels injected. Suggests immediate Senior Tech Lead overrides. |

## 🚀 Why These Factors? & Future Roadmap

**Why these 6?** We exhaustively researched enterprise post-mortem patterns. Nearly every catastrophic production deployment historically maps to a combination of: (A) Massive code drops, (B) Unfamiliar developers touching critical monoliths, or (C) Unseen architectural side-effects blowing up implicit dependencies. PRISM captures all three dimensions synchronously.

**Future 2.0 Roadmap (Predictive ML):** 
While base categorizations must remain deterministic, future versions of PRISM will extract incident-management MTTR data natively from Jira/PagerDuty. We will deploy localized Random Forest models to dynamically adjust the `Factor 3` Core Module weights based precisely on which components mathematically cause the most monetary outages locally within your specific organization.
